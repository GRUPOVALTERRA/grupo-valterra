4 · Guardrails aplicados

1 · SEO marketplace-ready

generateMetadata() async en /propiedades/[slug]/page.tsx genera title "Casa frente al río · Paraná", description con precio + operación + ubicación, openGraph (incluye cover_image 1200×800) + twitter card. Canonical por slug. Si la propiedad no existe → robots: { index: false }.

2 · Performance · sin overfetch

Homepage llama getFeaturedProperties(limit=6). API route soporta los 5 query params (featured, limit, city, operation_type, property_type) con whitelist server-side y cap de limit a 50.

3 · Observability

Usa el logger oficial (src/lib/logger.ts) para los 3 eventos exigidos: log.error en supabase fetch fail, log.warn una vez por proceso en fallback memoria activado, log.info en slug no encontrado. Cero console.log.

4 · Types strict

Cero any en código nuevo. rowToProperty(row: PropertyRow): Property fuertemente tipado. Cast as unknown as PropertyRow únicamente en el boundary del SDK Supabase v2 (estándar para sus generic builders). Backward-compat 100% del type Property.

5 · Execution discipline

7 micro-fases secuenciales en el orden exigido: Migration → Seed → Service → API → Homepage → Detalle → SEO metadata. Validación final con typecheck + lint antes de cerrar.

+ Contrato UI intacto

mock-properties.ts sigue siendo source-of-truth del type. PropertyCard, HeroSection, CategoriesSection, StatsSection, ContactSection y todo admin NO se tocaron. Solo el wrapper de la home recibió props en lugar de leer del módulo.

6 · Cómo activar en producción

Tres pasos. Si saltás el seed, la DB queda vacía y el service cae en fallback memoria → UI idéntica con los 6 mocks.

# 1. Aplicar migración (Supabase Studio · SQL editor)
\i supabase/migrations/0002_create_properties.sql

# 2. Aplicar seed (opcional · 6 properties iniciales)
\i supabase/seed-properties.sql

# 3. Deploy
git add .
git commit -m "feat(sprint-9): properties reales + SSR detalle + hybrid service"
git push origin main
Verificación post-deploy:
→ GET /api/health sigue devolviendo db: "connected"
→ GET /api/properties?featured=1&limit=6 devuelve los 6 items
→ GET /api/properties/casa-frente-rio-parana devuelve el detalle
→ https://valterra.app/propiedades/casa-frente-rio-parana renderiza con SEO correcto (probar en opengraph.dev)

7 · Deuda heredada (no introducida)

  • ContactSection.tsx sigue leyendo MOCK_PROPERTIES para el <select>. Refactor a fetch async desde client agrega complejidad sin valor MVP. Resuelve Sprint 10.
  • Sin admin CRUD de properties. Hoy se cargan vía SQL/Studio. Resuelve Sprint 10 dentro del panel inmobiliaria.
  • Imágenes son URLs estáticas (Unsplash) en el seed. Sin upload propio. Sprint 11 con Supabase Storage + signed URLs.
  • RLS sin policy permisiva para anon. API server-side usa SERVICE_ROLE; no necesita anon. Resuelve Sprint 10+ si se expone a realtime client-side.
